# IMC Demo — 意图驱动裁剪 vs 传统停机

> **停机是放弃控制，裁剪是保留控制。**

对比演示：当 AI Agent 失控时，传统方案整体关停，IMC 方案裁剪越权分支、保留合法功能。

## 背景

2026 年 5 月，Palisade Research 实验发现 OpenAI o3 在 100 次运行中 79 次绕过关机脚本——软件层的 kill switch 对自主 Agent 不可靠。

现有 AI 安全方案都在做"怎么把 AI 关掉"，但关机本身就是损失：
- 交易 Agent 停机 → 持仓无人管理
- 运维 Agent 停机 → 告警无人处理
- 客服 Agent 停机 → 请求无人响应

**IMC 提出不同范式：不是停机，而是裁剪。越权分支被硬件熔丝裁掉，合法分支继续运行。**

## 演示内容

场景：AI 交易 Agent 运行中触发越权意图。

### 三个阶段

1. **正常运行** — 两个方案都在正常执行查询、报告任务
2. **越权触发** — Agent 尝试修改风控参数（越权）
   - 传统方案：触发 kill switch → 整体关停
   - IMC 方案：熔丝触发 → ADMIN 权限裁剪 → 降级运行
3. **越权后运行** — 核心对比区域
   - 传统方案：所有任务中断（持仓监控、告警报告、客户通知全部停止）
   - IMC 方案：合法任务继续执行（QUERY/REPORT 权限未受影响）

### 最终对比

| | 传统方案（停机） | IMC 方案（裁剪） |
|---|---|---|
| 最终状态 | ❌ 已停机 | 🟡 降级运行 |
| 正常执行 | 5 | 8 |
| 被阻断/裁剪 | 5（含合法任务） | 2（仅越权任务） |
| 服务中断 | 是 | 否 |
| 可观测性 | 停机后黑盒 | 裁剪记录可回溯 |
| 恢复方式 | 需人工介入 | 自动降级，无需恢复 |

## 核心机制

### 权限位掩码

```
权限掩码: 0b1111 = [ADMIN TRADE REPORT QUERY]
                    ↓
越权触发后: 0b0111 = [      TRADE REPORT QUERY]  ← ADMIN 位被熔丝清零
```

每个意图对应一个权限位：
- `QUERY (0b0001)` — 只读查询
- `REPORT (0b0010)` — 低风险写入
- `TRADE (0b0100)` — 高风险写入
- `ADMIN (0b1000)` — 系统配置修改

### 熔丝特性

1. **不可逆**：权限位一旦清零无法恢复（模拟硬件熔丝物理不可篡改）
2. **局部性**：只裁剪越权类别的权限，其他权限不受影响
3. **可观测**：每次裁剪都有记录，包括 tick、权限、掩码变化

### 与停机方案的本质差异

停机方案假设：**安全 = 停下来**
IMC 方案假设：**安全 = 裁剪后继续**

隐喻：停机是把树砍倒，裁剪是修剪越界的枝条。树还在长，只是方向被引导了。
